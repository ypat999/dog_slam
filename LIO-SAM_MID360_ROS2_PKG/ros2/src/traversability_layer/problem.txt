## `traversability_layer` 模块分析报告

---

### 一、功能概述

这是一个 **Nav2 Costmap2D 插件层**，作为 `nav2_costmap_2d::CostmapLayer` 的子类实现，核心功能是：

1. **订阅 3D 点云**（默认话题 `/cloud_registered_body`）
2. **将点云投影到 2D 网格**，统计每个网格单元的高度信息（min_z, max_z, mean_z）
3. **计算坡度/梯度**：通过相邻网格的高度差计算 x/y 方向的斜率和总坡度
4. **根据两个指标计算通行代价**：
   - **台阶高度（step height）**：单个网格内 `max_z - min_z > step_height_threshold`（默认 0.15m）则标记为障碍
   - **坡度角（slope angle）**：坡度角超过 `max_slope_traversable`（默认 45°）则标记为致命障碍
5. **将代价写入 Nav2 的 master costmap**，供路径规划器避障使用

**设计意图**：为四足机器人在非平坦地形（台阶、斜坡等）导航时提供三维通行性分析，这是 2D 激光雷达无法做到的。

---

### 二、使用方式

#### 1. 编译集成
在 `nav2_params_sim.yaml` 中已经配置了该插件，位于 local_costmap 的插件链中：

```yaml
plugins: ["stvl_layer", "obstacle_layer_local", "traversability_layer", "inflation_layer_local"]

traversability_layer:
  plugin: "traversability_layer::TraversabilityLayer"
  enabled: true
  pointcloud_topic: "/cloud_registered"
  step_height_threshold: 0.15      # 台阶高度阈值
  max_slope_traversable: 45.0      # 最大可通行坡度(度)
  slope_cost_scale: 2.0            # 坡度代价缩放
  height_cost_scale: 3.0           # 高度差代价缩放
  ...
```

#### 2. 可调参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `pointcloud_topic` | `/cloud_registered_body` | 点云话题 |
| `step_height_threshold` | 0.15m | 网格内高度差超过此值视为台阶障碍 |
| `max_slope_traversable` | 45° | 最大可通行坡度 |
| `slope_cost_scale` | 2.0 | 坡度代价权重 |
| `height_cost_scale` | 3.0 | 高度差代价权重 |
| `publish_slope_map` | false | 是否发布坡度可视化点云 |

---

### 三、能否起作用？—— **基本框架可用，但有多个严重问题**

整体架构是合理的，核心逻辑（点云→高度图→坡度→代价）的思路正确。但存在以下问题，**不修复的话效果会很差甚至无法正常工作**：

---

### 四、详细问题分析

#### 🔴 严重问题

**1. 每帧点云都完全重置高度图（[traversability_layer.cpp:188-200](file:///home/ywj/dog_slam/LIO-SAM_MID360_ROS2_PKG/ros2/src/traversability_layer/src/traversability_layer.cpp#L188-200)）**

```cpp
for (auto & cell : height_map_) {
    cell.min_z = std::numeric_limits<float>::max();
    cell.max_z = std::numeric_limits<float>::lowest();
    cell.mean_z = 0.0f;
    cell.point_count = 0;
    ...
    cell.has_data = false;
}
```

每次收到点云回调，**整个高度图被清零重建**。这意味着：
- 历史信息全部丢失——如果某帧点云没有覆盖某个区域，该区域变为"无数据"
- 对于 rolling window 的 local costmap，机器人移动后很多区域会频繁丢失数据
- **正确做法**：应该累加/滑动窗口更新，或者至少不清除仍在视野内的区域

**2. `updateBounds` 只扩展有障碍的格子边界（[traversability_layer.cpp:316-337](file:///home/ywj/dog_slam/LIO-SAM_MID360_ROS2_PKG/ros2/src/traversability_layer/src/traversability_layer.cpp#L316-337)）**

```cpp
if (cell.height_diff > step_height_threshold_ ||
    cell.slope_magnitude > std::tan(max_slope_traversable_)) {
    *min_x = std::min(*min_x, wx - res);
    ...
}
```

`updateBounds` 只在有障碍物时扩展边界，**没有数据的区域不会被标记**。这导致：
- 之前标记为障碍的区域，如果新帧没有覆盖，障碍信息会消失（因为高度图被重置了）
- `updateBounds` 不扩展整个有数据的区域，可能导致 `updateCosts` 的有效范围不完整

**3. `computeCost` 逻辑矛盾（[traversability_layer.cpp:274-286](file:///home/ywj/dog_slam/LIO-SAM_MID360_ROS2_PKG/ros2/src/traversability_layer/src/traversability_layer.cpp#L274-286)）**

```cpp
if (cell.height_diff > step_height_threshold_) {
    float height_excess = cell.height_diff - step_height_threshold_;
    float height_cost = height_excess * height_cost_scale_ * 100.0f;
    ...
    return static_cast<unsigned char>(...height_cost);  // 可能返回很小的值
}
```

当 `height_diff` 刚刚超过阈值（如 0.16m），`height_excess = 0.01`，`height_cost = 0.01 * 3.0 * 100 = 3`，返回代价仅为 **3**（几乎等于自由空间）。这意味着一个 0.16m 的台阶只被标记为代价 3，完全不会阻止机器人通过。**应该直接返回 LETHAL_OBSTACLE 或至少是一个高代价值**。

**4. 点云话题名称不一致**

- 代码默认值：`/cloud_registered_body`
- sim 配置中：`/cloud_registered`
- 实际 LIO-SAM 发布的话题需要确认

如果话题名不匹配，整个层不会收到任何数据。

#### 🟡 中等问题

**5. TF 超时时间过长（[traversability_layer.cpp:121](file:///home/ywj/dog_slam/LIO-SAM_MID360_ROS2_PKG/ros2/src/traversability_layer/src/traversability_layer.cpp#L121)）**

```cpp
transform = tf_buffer_->lookupTransform(
    target_frame, source_frame, msg->header.stamp,
    rclcpp::Duration::from_seconds(0.5));  // 500ms 超时
```

在 costmap 的回调中阻塞等待 TF 500ms，会严重拖慢 costmap 更新频率。Nav2 内置的 obstacle layer 通常使用更短的超时或异步方式。

**6. 手动旋转矩阵计算（[traversability_layer.cpp:130-148](file:///home/ywj/dog_slam/LIO-SAM_MID360_ROS2_PKG/ros2/src/traversability_layer/src/traversability_layer.cpp#L130-148)）**

手动从四元数计算旋转矩阵进行点云变换，而不是使用 `tf2::doTransform` 或 PCL 的变换函数。虽然功能上可以工作，但：
- 代码冗长易出错
- 没有利用已有的库函数
- 如果 TF 树中存在缩放，这里会忽略

**7. `updateCosts` 中的合并策略过于简单（[traversability_layer.cpp:356-362](file:///home/ywj/dog_slam/LIO-SAM_MID360_ROS2_PKG/ros2/src/traversability_layer/src/traversability_layer.cpp#L356-362)）**

```cpp
if (old_cost == nav2_costmap_2d::NO_INFORMATION || cost > old_cost) {
    master_array[idx] = cost;
}
```

只取最大值，没有考虑与其他层（如 stvl_layer、obstacle_layer）的交互。特别是当 traversability_layer 标记了一个区域为低代价（如上面问题3所述），可能会覆盖其他层正确标记的高代价。

**8. 缺少 `matchSize` 实现**

作为 `CostmapLayer` 的子类，通常需要重写 `matchSize()` 方法来在 costmap 尺寸变化时同步内部数据结构。当前代码在 `processCloud` 中动态调整 `height_map_` 大小，但没有在 `onInitialize` 中调用 `matchSize()`，可能导致首次更新前尺寸不匹配。

#### 🟢 小问题

**9. `slope_magnitude` 与 `tan(slope_angle)` 混用**

- `computeSlope` 中计算 `slope_magnitude = sqrt(dz_dx² + dz_dy²)`，这是**梯度的模**，等于 `tan(slope_angle)`
- `updateBounds` 中用 `cell.slope_magnitude > std::tan(max_slope_traversable_)` 比较——这是正确的
- `computeCost` 中用 `std::atan(cell.slope_magnitude)` 转回角度——也是正确的
- 但 `max_slope_traversable_` 在 `onInitialize` 中被转换为弧度，而 `slope_magnitude` 是 tan 值，两者物理含义不同，容易混淆

**10. 没有处理 rolling window 滚动时的坐标偏移**

对于 rolling window 的 local costmap，当机器人移动时，costmap 的 origin 会变化。当前代码直接用 `getOriginX/Y()` 和 `getResolution()` 来映射点云，这在单帧内是正确的，但由于每帧都重置高度图（问题1），历史数据丢失，所以滚动窗口的效果很差。

## 一、连续台阶代价计算分析

这是最关键的问题，需要仔细推演。假设一个典型楼梯场景：

### 场景推演

假设：网格分辨率 0.05m，台阶高度 0.15m，台阶深度 0.3m（约6个格子宽）

```
侧视图（楼梯）:
    ████  第3级 (z=0.45m)
    ████
      ████  第2级 (z=0.30m)
      ████
        ████  第1级 (z=0.15m)
        ████
          ████  地面 (z=0.00m)
```

**当前代码对每个格子独立计算 `height_diff = max_z - min_z`**：

| 格子位置 | 包含的点 | height_diff | 判定 |
|----------|---------|-------------|------|
| 台阶平面上的格子 | 只有同一高度的点 | ≈ 0 | ✅ 自由 |
| 台阶边缘的格子 | 可能同时有上、下两级高度的点 | ≈ 0.15m | ⚠️ 刚好到阈值 |

**关键发现**：台阶边缘的格子只有1-2个格子宽（0.05-0.10m），因为点云在台阶立面处通常没有点（LiDAR射线平行于立面时打不到）。所以：

1. **每个台阶的代价确实是独立计算的**——台阶平面是自由的，只有极窄的边缘有代价
2. **对于四足机器人，这其实是合理的行为**——0.15m的台阶本来就应该可以通行
3. **但问题在于**：如果 `step_height_threshold = 0.15m`，那么 0.16m 的台阶边缘格子 `height_diff = 0.16`，按当前代码代价仅为 `(0.01 * 3.0 * 100) = 3`，几乎等于自由空间——**这才是真正的bug**

### 连续台阶 vs 单个高台阶的区别

- **连续楼梯**：每级0.15m，每级台阶平面是自由的 → 机器人可以逐级攀爬 → ✅ 正确
- **0.5m高墙**：单个格子 `height_diff = 0.5m`，远超阈值 → 应该是致命障碍 → ❌ 当前代码只给低代价
- **斜坡**：每个格子 `height_diff` 很小，但 `slope_magnitude` 大 → 由坡度判断 → ✅ 合理

**结论**：逐格子独立计算的设计本身是正确的，问题出在代价曲线的形状——超过阈值后代价上升太慢。

## 二、修正方案

### 修改1：高度图持久化 + Raycast 清除（核心改动）

**问题**：当前每帧完全重置高度图，导致历史信息丢失。

**方案**：

```
CellData 增加字段:
  - last_seen_time: rclcpp::Time  // 最后观测时间
  - is_cleared: bool               // 是否被raycast清除

处理流程:
  1. 新点云到达 → 获取传感器原点在costmap坐标系中的位置
  2. Raycast清除:
     - 对每个观测到的点，从传感器原点到该点画射线
     - 射线经过的格子：标记为"已清除"（该方向无障碍）
     - 射线终点的格子：更新高度信息
  3. 合并更新:
     - 被raycast清除的格子：重置该格子的min_z/max_z，用新点云重新填充
     - 未被raycast扫过且未被新点云覆盖的格子：保留旧数据
  4. 时间衰减:
     - 超过 max_update_duration（如5秒）未被观测的格子：逐渐降低置信度
     - 超过 clearing_duration（如10秒）：标记为NO_INFORMATION
```

**关于雷达倾斜**：由于雷达可能倾斜，传感器原点在costmap坐标系中的Z值不为0。Raycast时需要：
- 将传感器原点投影到2D costmap平面
- 从2D投影点向每个点的2D投影画射线
- 清除射线路径上的格子（因为射线能穿过说明该方向无遮挡）

这和 Nav2 VoxelLayer 的 raycast 逻辑类似，但简化为2D。

### 修改2：代价计算——指数增长曲线

**问题**：当前线性增长，超过阈值后代价太低。

**方案**：参考 InflationLayer 的 `exp(-cost_scaling_factor * distance)` 思路，反过来用：

```cpp
unsigned char computeCost(const CellData & cell) const
{
    if (!cell.has_data) return NO_INFORMATION;
    
    // === 台阶高度代价 ===
    unsigned char height_cost = 0;
    if (cell.height_diff > step_height_threshold_) {
        float excess = cell.height_diff - step_height_threshold_;
        // 指数增长: cost = 254 * (1 - exp(-scale * excess))
        // excess=0时 cost=0, excess增大时 cost快速逼近254
        float ratio = 1.0f - std::exp(-height_cost_scale_ * excess);
        height_cost = static_cast<unsigned char>(
            std::min(254.0f, ratio * 254.0f));
    }
    
    // === 坡度代价 ===
    unsigned char slope_cost = 0;
    float slope_angle = std::atan(cell.slope_magnitude);
    if (slope_angle > max_slope_traversable_) {
        slope_cost = LETHAL_OBSTACLE;
    } else if (slope_angle > slope_cost_start_) {  // 新参数: 开始产生代价的坡度角
        float excess_angle = slope_angle - slope_cost_start_;
        float max_excess = max_slope_traversable_ - slope_cost_start_;
        float ratio = excess_angle / max_excess;
        // 指数增长
        float cost_f = (1.0f - std::exp(-slope_cost_scale_ * ratio)) * 254.0f;
        slope_cost = static_cast<unsigned char>(std::min(254.0f, cost_f));
    }
    
    // 取两者最大值
    return std::max(height_cost, slope_cost);
}
```

**效果对比**（以 height_cost_scale_=10.0 为例）：

| 超出阈值 | 当前代码代价 | 新方案代价 |
|----------|------------|-----------|
| 0.01m | 3 | 24 |
| 0.05m | 15 | 96 |
| 0.10m | 30 | 168 |
| 0.20m | 60 | 228 |
| 0.30m | 90 | 245 |
| 0.50m | 150 | 253 |

指数曲线在超出阈值后急剧上升，0.05m 的超出就已经接近 LETHAL。

### 修改3：新增参数


### 修改4：Raycast 清除实现思路

```
raycastClear(sensor_origin_2d, observed_points_2d):
  1. 对每个观测点 (px, py):
     a. 计算从 sensor_origin 到 (px,py) 的射线方向
     b. 用 Bresenham 算法遍历射线经过的所有格子
     c. 对经过的格子: 标记为 "ray_passed"
  2. 对所有 ray_passed 的格子:
     - 如果该格子本帧没有被新点云标记 → 重置该格子
     - 这意味着: 射线能穿过去, 说明该方向没有障碍
  3. 对射线终点的格子:
     - 用新点云数据更新高度信息
```

**注意**：raycast 只清除传感器"看穿"的格子。传感器看不到的区域（遮挡后方）保留旧数据，直到超时衰减。

### 修改5：updateBounds 修正

当前 `updateBounds` 只扩展有障碍的格子。应该改为：**扩展所有有数据的格子**，因为低代价区域也需要写入 master costmap（否则其他层可能误标记）。

### 修改6：TF 超时

保持 0.5s 不变（用户说可以放宽），但建议增加 TF 超时的日志级别，从 WARN 降为 DEBUG，避免刷屏。

## 四、关于连续台阶的最终结论

**每个台阶确实是独立计算代价的**，这是正确的设计：

1. **0.15m 的连续楼梯**：每级台阶平面自由，边缘极窄代价带 → 机器人可以通行 → ✅
2. **0.3m 的大台阶**：边缘格子 `height_diff=0.3m`，超出阈值 0.15m，指数曲线下代价 ≈ 228 → 接近致命 → ❌ 不可通行 → ✅
3. **斜坡**：每个格子 `height_diff` 小，但 `slope_magnitude` 大 → 由坡度代价控制 → ✅

**唯一需要注意的是**：如果台阶边缘的点云恰好只打到一个面（上或下），则该格子 `height_diff ≈ 0`，不会产生代价。这是 LiDAR 的固有限制，不是代码问题。实际使用中，台阶立面附近的格子通常会有相邻格子的坡度代价来补充。
